Необходимо реализовать в системе статусную модель декларации.

Контекст:
В системе декларация может быть создана тремя способами:
1. вручную через интерфейс,
2. автоматически после загрузки документов через Telegram-бот,
3. автоматически после загрузки документов по email.

Важно:
Нужно разделить:
- основной статус декларации,
- состояние обработки документов,
- признак подписи.

Не смешивать эти сущности в одно поле.

==================================================
1. ОСНОВНОЙ СТАТУС ДЕКЛАРАЦИИ
==================================================

Основной статус отражает этап работы пользователя с декларацией.

Использовать следующие значения:

1. NEW
Отображаемое название: "Новая"

Описание:
Декларация создана в системе, но пользователь еще не начал работу с ней в интерфейсе.

В этот статус попадают:
- декларации, созданные вручную через интерфейс;
- декларации, созданные через Telegram-бот;
- декларации, созданные через email.

Важно:
Статус "Новая" не означает, что декларация обязательно уже заполнена или обязательно пустая.
Смысл статуса:
декларация создана, но пользователь еще не начал работать с ней в интерфейсе.

2. REQUIRES_ATTENTION
Отображаемое название: "Требует внимания"

Описание:
Пользователь уже открыл декларацию, и в ней есть проблемы:
- незаполненные обязательные поля,
- конфликты значений,
- ошибки валидации,
- недостаток обязательных документов,
- другие блокирующие или значимые замечания.

3. READY_TO_SEND
Отображаемое название: "Готово к отправке"

Описание:
Пользователь открыл декларацию, и в ней нет блокирующих проблем.
Декларация готова к подписанию и отправке.

4. SENT
Отображаемое название: "Отправлено"

Описание:
Декларация отправлена в таможню.

==================================================
2. СОСТОЯНИЕ ОБРАБОТКИ ДОКУМЕНТОВ
==================================================

Это отдельное поле, не связанное напрямую с основным статусом.

Использовать следующие значения:

1. NOT_STARTED
Отображаемое название: "Распознавание не запускалось"

Описание:
Документы еще не обрабатывались системой.

Типичный сценарий:
Пользователь создал декларацию вручную через интерфейс, но не запускал распознавание.

2. PROCESSING
Отображаемое название: "В обработке"

Описание:
Система выполняет распознавание документов, извлечение данных и автозаполнение декларации.

3. AUTO_FILLED
Отображаемое название: "Заполнена автоматически"

Описание:
Система успешно обработала документы и заполнила декларацию.

4. PROCESSING_ERROR
Отображаемое название: "Ошибка обработки"

Описание:
Во время распознавания или извлечения данных произошла ошибка.

Опционально можно предусмотреть PARTIALLY_FILLED / "Заполнена частично", но для MVP не обязательно.

==================================================
3. ПРИЗНАК ПОДПИСИ
==================================================

Это отдельное поле.

Использовать следующие значения:

1. UNSIGNED
Отображаемое название: "Не подписана"

2. SIGNED
Отображаемое название: "Подписана"

Важно:
Подпись не должна быть частью основного статуса.
Например:
декларация может быть в статусе READY_TO_SEND и при этом быть UNSIGNED или SIGNED.

==================================================
4. ЛОГИКА СОЗДАНИЯ ДЕКЛАРАЦИИ
==================================================

Сценарий 1. Создание вручную через интерфейс
- пользователь создает декларацию;
- основной статус = NEW;
- состояние обработки = NOT_STARTED;
- подпись = UNSIGNED.

Сценарий 2. Создание через Telegram-бот
- пользователь отправляет пакет документов;
- система создает декларацию;
- основной статус = NEW;
- состояние обработки сначала = PROCESSING;
- после успешной обработки = AUTO_FILLED;
- если ошибка = PROCESSING_ERROR;
- подпись = UNSIGNED.

Сценарий 3. Создание через email
- пользователь отправляет документы по почте;
- система создает декларацию;
- основной статус = NEW;
- состояние обработки сначала = PROCESSING;
- после успешной обработки = AUTO_FILLED;
- если ошибка = PROCESSING_ERROR;
- подпись = UNSIGNED.

==================================================
5. ПЕРЕХОДЫ ОСНОВНОГО СТАТУСА
==================================================

1. Любая новая декларация при создании получает:
status = NEW

2. При первом открытии декларации пользователем:
если найдены проблемы:
status: NEW -> REQUIRES_ATTENTION

если проблем нет:
status: NEW -> READY_TO_SEND

3. Если декларация уже в статусе REQUIRES_ATTENTION и все проблемы исправлены:
status: REQUIRES_ATTENTION -> READY_TO_SEND

4. Если декларация в статусе READY_TO_SEND, но после изменений снова появились проблемы:
status: READY_TO_SEND -> REQUIRES_ATTENTION

5. Если декларация в статусе READY_TO_SEND и успешно отправлена:
status: READY_TO_SEND -> SENT

Важно:
Отправка возможна только если:
- status = READY_TO_SEND
- signatureStatus = SIGNED

==================================================
6. ПЕРЕХОДЫ СОСТОЯНИЯ ОБРАБОТКИ
==================================================

1. NOT_STARTED -> PROCESSING
Когда пользователь запускает распознавание документов.

2. PROCESSING -> AUTO_FILLED
Когда обработка завершилась успешно.

3. PROCESSING -> PROCESSING_ERROR
Когда обработка завершилась с ошибкой.

4. PROCESSING_ERROR -> PROCESSING
При повторном запуске обработки.

5. AUTO_FILLED -> PROCESSING
Если пользователь заменил документы и заново запустил обработку.

==================================================
7. ПЕРЕХОДЫ ПОДПИСИ
==================================================

1. UNSIGNED -> SIGNED
Когда пользователь выполняет действие "Подписать декларацию".

2. SIGNED -> UNSIGNED
Подпись должна автоматически сбрасываться, если после подписания произошло любое значимое изменение декларации.

Под значимыми изменениями понимать:
- изменение любого значимого поля декларации;
- выбор другого значения для поля;
- изменение источника значения;
- замена документа;
- добавление нового документа, влияющего на заполнение;
- удаление документа;
- повторное автозаполнение / пересборка данных;
- повторная генерация структуры данных, влияющей на содержимое декларации.

==================================================
8. БИЗНЕС-ПРАВИЛА
==================================================

1. Основной статус, состояние обработки и признак подписи должны храниться отдельно.

2. Статус NEW должен использоваться для всех деклараций до первого входа пользователя в декларацию, независимо от канала создания.

3. Если декларация создана вручную и распознавание не запускалось:
- status = NEW
- processingStatus = NOT_STARTED

4. Если декларация создана через Telegram/email и обработка завершилась успешно, но пользователь ее еще не открывал:
- status = NEW
- processingStatus = AUTO_FILLED

5. Если пользователь впервые открыл декларацию:
- нужно выполнить проверку готовности;
- если есть хотя бы одна проблема, присвоить REQUIRES_ATTENTION;
- если проблем нет, присвоить READY_TO_SEND.

6. Отправка декларации должна быть запрещена, если:
- status != READY_TO_SEND
или
- signatureStatus != SIGNED

7. После любого значимого изменения подписанной декларации:
- signatureStatus автоматически меняется на UNSIGNED;
- далее нужно повторно проверить декларацию;
- если появились проблемы, основной статус должен стать REQUIRES_ATTENTION;
- если проблем нет, статус может остаться READY_TO_SEND.

==================================================
9. ЧТО НУЖНО РЕАЛИЗОВАТЬ
==================================================

Нужно:
1. Описать enum / constants для:
- declaration status
- processing status
- signature status

2. Реализовать сервис / доменную логику переходов статусов.

3. Реализовать единый метод/функцию, например:
recalculateDeclarationState(declaration)
который:
- анализирует декларацию,
- определяет наличие проблем,
- обновляет основной статус,
- при необходимости сбрасывает подпись.

4. Реализовать guard / validation для отправки:
отправка возможна только при READY_TO_SEND + SIGNED.

5. Реализовать читаемые display labels для UI.

6. По возможности не размазывать логику по интерфейсу:
основные правила должны жить в доменном слое / сервисе состояния декларации.

==================================================
10. ЖЕЛАЕМЫЙ РЕЗУЛЬТАТ
==================================================

Нужно спроектировать и реализовать чистую, расширяемую статусную модель декларации, в которой:
- основной статус отражает этап работы пользователя,
- состояние обработки отражает статус AI-обработки документов,
- подпись учитывается отдельно,
- переходы управляются прозрачными бизнес-правилами.
