Перестрой экран страницы декларации в системе цифрового брокера под новую UX-концепцию:
Главная идея
Это не экран тотальной проверки всех полей после AI.
Это ready-to-send dashboard / review by exception interface.
Система почти всегда корректно заполняет декларацию автоматически.
Пользователь не должен проверять всю декларацию целиком.
Он должен:
сразу увидеть, готова ли декларация к отправке,
увидеть только исключения и проблемные места,
получить краткую сводку по кейсу,
при необходимости открыть полную декларацию,
точечно скорректировать отдельные поля,
выгрузить PDF / XML,
подписать ЭЦП,
отправить декларацию в таможню.
UX-подход:
default = ready to send
fallback = inspect and correct
Что нужно изменить в существующей концепции
Убери ощущение, что пользователь обязан проходить всю декларацию и проверять каждое поле после AI.
Не делай экран похожим на огромную editable-форму.
Не делай источник данных постоянно открытым и обязательным элементом интерфейса.
Не делай акцент на “AI заполнил поле — теперь пользователь должен это проверить”.
Вместо этого сделай акцент на:
статус готовности,
исключения,
быстрый выпуск декларации,
точечную корректировку по запросу,
полную детализацию только on demand.
Layout
Создай desktop enterprise screen, ширина 1440–1600 px.
Экран должен быть светлым, современным, premium B2B SaaS, спокойным, высокоплотным, профессиональным.
Структура экрана:
Sticky top header
Ready status hero card
Two-column main content
left: exceptions / issues requiring attention
right: declaration summary
Documents summary section
Auto-validation section
Quick release actions
Optional detailed access links
open full declaration
open documents
open history
open validation details
1. Sticky top header
Верхняя панель должна быть очень чистой, enterprise-level, уверенной.
Слева:
back button
case id: DC-2026-001245
client: ООО Альфа Импорт
По центру:
overall status badge:
вариант 1: Готово к отправке
вариант 2: Готово, но есть замечания
вариант 3: Требует действий
last update timestamp
Справа:
Actions:
PDF
XML
Подписать ЭЦП
Отправить
Если кейс готов — основной акцент на Подписать и отправить
Если есть замечания — основной акцент на Открыть замечания
Если есть блокеры — основной акцент на Исправить проблемы
Сделай CTA сценарным, зависящим от статуса.
2. Ready status hero card
Это главный блок экрана.
Он должен сразу отвечать пользователю: можно отправлять или нет.
Сделай большую premium status card.
Пример состояния:
Декларация готова к отправке
Покажи в карточке:
Все обязательные поля заполнены
Комплект документов полный
XML валиден
PDF сформирован
Критических ошибок: 0
Предупреждений: 2
Кнопки внутри карточки:
Primary:
Подписать и отправить
Secondary:
Открыть замечания
Открыть декларацию
Альтернативный сценарий:
Если есть blocking issues, status card должна меняться на:
Декларация не готова к отправке
2 критические ошибки блокируют выпуск
Primary CTA: Исправить проблемы
Сделай этот блок визуально самым важным на странице.
3. Main two-column section
Left column: “Что требует внимания”
Это главный рабочий блок.
Не показывай всю декларацию.
Покажи только исключения.
Раздели исключения на 3 группы:
A. Blocking
Например:
Графа 20. Условия поставки не заполнена
helper text: Значение не найдено в загруженных документах
actions:
Найти
Ввести вручную
Открыть декларацию
B. Review recommended
Например:
Товар 3 — низкая уверенность по коду ТН ВЭД
helper text: AI уверен на 68%
actions:
Посмотреть источник
Выбрать альтернативу
Подтвердить
Графа 22 — конфликт между инвойсом и контрактом
helper text: Найдены 2 возможных значения
actions:
Сравнить
Выбрать источник
C. Info
Например:
5 полей были заполнены вручную
Использована новая версия инвойса
Добавлен сертификат происхождения
Для info-сообщений нужны более спокойные карточки/rows.
Если blocking issues нет — покажи красивое сообщение:
Критических проблем не обнаружено
Если warnings нет — покажи:
Дополнительная проверка не требуется
Right column: “Краткая сводка декларации”
Это compact summary card, чтобы пользователь быстро понял, что система собрала правильный кейс.
Покажи:
тип процедуры: ИМ 40
декларант: ООО Альфа Импорт
отправитель: Shanghai Industrial Co.
получатель: ООО Альфа Импорт
инвойс: INV-2026-101
контракт: CTR-55/26
условия поставки: FOB
валюта: USD
сумма: 12 540,00
товарных позиций: 12
вес брутто: 2 340 кг
страна отправления: Китай
Сделай это как premium information summary card, не как форма.
4. Documents summary section
Ниже главного блока сделай спокойную секцию документов.
Это не должен быть огромный левый sidebar со всеми деталями.
На главном экране нужен documents summary, а не document review workspace.
Покажи:
Загружено документов: 5
Обязательные документы: 4/4
Дополнительные: 1
Проблемы качества: 1
Ниже — компактные document chips / rows:
Инвойс — OK
Контракт — OK
Упаковочный лист — OK
Транспортный документ — OK
Сертификат происхождения — не требуется / загружен / отсутствует
Добавь secondary action:
Открыть документы
5. Auto-validation section
Добавь отдельную спокойную секцию:
Автопроверка завершена
Покажи checklist summary:
[✓] Все обязательные поля заполнены
[✓] Комплект документов полный
[✓] XML валиден
[✓] Расчетные проверки пройдены
[!] Есть 2 предупреждения
Ниже secondary link/button:
Показать детали проверки
Эта секция должна транслировать, что система уже сама сделала основную работу по валидации.
6. Quick release actions section
Ниже добавь блок с основными действиями выпуска.
Если все готово:
main CTA: Подписать и отправить
Рядом secondary actions:
Скачать PDF
Скачать XML
Открыть декларацию
Открыть замечания
Если есть блокеры:
main CTA: Исправить проблемы
Этот блок должен быть аккуратным и сценарным, не просто набором случайных кнопок.
7. Optional details / secondary navigation
Внизу страницы или как secondary links добавь доступ к деталям:
Открыть полную декларацию
Открыть документы
История изменений
Детали проверок
Важно: эти действия должны выглядеть как secondary / optional.
Они нужны, но не должны перетягивать внимание с главного сценария выпуска.
8. Full declaration access model
В макете покажи, что полная декларация — это вторичный режим, а не главный экран.
Можно визуально намекнуть на это через:
secondary button Открыть декларацию
маленький preview-card ниже
или ссылку в optional details block
Полная декларация должна восприниматься как:
доступна по запросу,
mostly read-only,
редактирование точечное,
просмотр источников по клику.
Не делай полную форму главным центром экрана.
9. Tone and visual style
Интерфейс должен транслировать:
высокий уровень автоматизации,
уверенность,
контроль,
юридическую значимость,
enterprise reliability,
premium operational efficiency.
Стиль:
light theme
soft neutral backgrounds
white / slightly tinted cards
subtle borders
rounded corners 14–18 px
soft shadows
compact typography
strong hierarchy
low visual noise
minimal but clear status colors
Цветовая логика:
ready / success: мягкий зеленый или нейтральный success
warning / review recommended: amber
blocking / error: red
info: blue-gray
AI presence should be subtle, not dominant
Важно:
Не делай интерфейс похожим на “AI playground”.
Это должен быть серьезный B2B интерфейс для выпуска таможенной декларации.
10. Components to emphasize
Особенно качественно проработай:
ready status hero card
issue cards with priority hierarchy
compact summary card
validation summary card
release action bar
document summary row/chips
scenario-based primary CTA
clean information architecture
11. Final outcome
Сгенерируй один high-fidelity desktop screen для страницы декларации в режиме:
ready-to-send dashboard / review by exception
Экран должен показывать:
декларация почти или полностью готова,
пользователь видит только то, что требует внимания,
полная декларация доступна как secondary action,
фокус на быстром выпуске,
визуально понятно, что AI выполнил основную работу,
пользователь включается только в исключениях.
