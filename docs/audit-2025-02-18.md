## Анализ невыполненных пунктов (4, 6, 7, 8, 9, 11, 16, 17)

### Пункт 4 — CounterpartyLookup (Frontend)

**Текущее состояние:** Backend готов (counterparties роутер), но нет UI-компонента.

**Логика для проекта:** Брокер каждый раз вводит одних и тех же контрагентов. CounterpartyLookup — автодополнение: начинаешь вводить "Shenzhen" → выпадает список с ИНН, КПП, адресом. Встраивается в DeclarationEditPage на шаге "Участники сделки". Экономит 3-5 минут на каждую декларацию.

### Пункт 6 — LlamaIndex RAG ядро

**Текущее состояние:** index_manager.py работает на голом ChromaDB (прямые col.query()). Простой cosine search.

**Логика для проекта:** LlamaIndex дал бы query engines с response synthesis, SubQuestionQueryEngine для мульти-индексных запросов, hybrid retrieval (keyword + vector). Но текущая связка "ChromaDB + LLM direct" уже закрывает 80% потребностей. LlamaIndex ранее вызывал бесконечный pip backtrack.

**Рекомендация:** можно отложить или закрыть as-is.

### Пункт 7 — DSPy модули (полноценно)

**Текущее состояние:** HSCodeClassifier и RiskAnalyzer реально используют DSPy. InvoiceExtractor, ContractExtractor, PackingExtractor — заглушки, всегда regex fallback. dspy_optimizer.py — не проверен.

**Логика для проекта:**
- Включить DSPy для InvoiceExtractor — LLM с оптимизированным промптом вместо regex
- Включить DSPy для PackingExtractor — убрать `if False:`
- dspy_optimizer.py — автооптимизация: при 50+ примеров DSPy BootstrapFewShot подбирает лучшие few-shot примеры

### Пункт 8 — CrewAI мультиагенты

**Текущее состояние:** agent_crew.py — 831 строка реального кода. Код CrewAI агентов написан, но crewai нет в requirements.txt → _crewai_available = False → метод _run_crewai() никогда не вызывается.

**Логика для проекта:** Текущий pipeline линейный и уже работает. CrewAI добавляет параллельное выполнение и "роли" агентов в промптах. Но это overhead — тяжёлая зависимость для небольшого выигрыша.

**Рекомендация:** можно отложить, pipeline работает.

### Пункт 9 — Phoenix observability

**Текущее состояние:** observability.py — 37 строк реального кода. Phoenix контейнер в docker-compose. Но arize-phoenix нет в requirements.txt.

**Логика для проекта:** Для прода критично: видеть каждый LLM-вызов, токены, стоимость, latency, обнаружение галлюцинаций. Dashboard на :6006.

**Что доделать:** добавить arize-phoenix и openinference-instrumentation-openai в requirements.txt.

### Пункт 11 — Open-RAG-Eval

**Текущее состояние:** Ничего.

**Логика для проекта:** Проверка качества RAG без ручной разметки: relevance, faithfulness, context utilization. Тестовый набор: PDF из dek/. Регрессионное тестирование после переиндексации.

### Пункт 16 — Cortex Memory

**Текущее состояние:** Прецеденты уже в ChromaDB (index_manager.add_precedent/search_precedents).

**Логика для проекта:** Cortex Memory — продвинутая версия: автоизвлечение фактов из деклараций, self-improving поиск, REST API. Текущие прецеденты в ChromaDB дают 70% этой функциональности.

**Рекомендация:** можно отложить.

### Пункт 17 — База знаний, чек-листы, admin

**Текущее состояние:** Ничего.

**Логика для проекта:**

## Ключевая проблема

Код для CrewAI, Phoenix и DSPy (парсеры) **написан**, но зависимости не в requirements.txt. При сборке Docker-образа эти библиотеки не устанавливаются и весь AI-стек работает через fallback (regex/LLM напрямую). Для активации достаточно добавить зависимости в requirements.txt и пересобрать ai-service.
