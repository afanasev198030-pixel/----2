# Roadmap: Enterprise Customs Automation Platform

Единый план развития платформы от текущего AI-парсера деклараций к полноценной экосистеме ВЭД: документы, комплаенс, финансы, логистика, интеграции и клиентский сервис.

## Как использовать этот документ

- Это мастер-план (стратегический уровень).
- Каждая фаза детализируется в отдельный план реализации (тактический уровень): `docs/plans/<phase>-<topic>.md`.
- Все новые feature-планы должны ссылаться на конкретный пункт этого roadmap.
- Изменения статусов фаз вносятся сначала в детальные планы, затем агрегируются сюда.

## Целевой продукт

Платформа автоматизации таможенного оформления, которая:
- принимает документы из разных каналов (web, email, bot),
- автоматически формирует и проверяет декларацию,
- снижает регуляторные и финансовые риски,
- поддерживает оплату услуг и юридически значимое подписание документов,
- добавляет смежные сервисы (логистика, интеграции с клиентскими системами),
- поддерживает контроль unit-экономики AI-обработки.

## Фазы развития

### Фаза 1: Умный Intake и AI-Стратегии (ближайший приоритет)

Цель: убрать необходимость ручного входа клиента в UI для старта процесса и сделать AI-логику управляемой через бизнес-правила.

#### 1.1 Email-Intake шлюз (Email-to-Declaration)

- Создать сервис, который слушает выделенный почтовый ящик (IMAP) или принимает webhook от почтового сервера.
- Автоматически извлекать вложения (PDF, Excel) из писем клиентов.
- Отправлять файлы в существующий `POST /api/v1/ai/parse-smart`.
- Автоматически создавать черновик декларации в БД и привязывать к клиенту по email отправителя.
- Технологии: `imaplib`/`aiosmtpd`, очередь задач (Celery).

#### 1.2 AI-стратегии (Global Rules Engine)

- Добавить UI в админке для текстовых стратегий заполнения (правила уровня бизнеса).
- Сохранять стратегии в БД (таблица `ai_strategies`).
- Интегрировать стратегии в `agent_crew.py` как часть системных инструкций при финальной компиляции декларации.
- Пример стратегии: "Если поставщик ZED Group, ставить EXW и пост Шереметьево".

#### 1.3 Связывание документов (Side-by-Side Viewer)

- Доработать `DeclarationEditPage.tsx`.
- Добавить split-view: слева форма декларации, справа PDF viewer (`iframe` или `react-pdf`).
- По клику на прикрепленный документ показывать его содержимое для визуальной сверки.

#### 1.4 Обучение подбора кода ТН ВЭД по факту оформления

- Добавить контур обучения на данных клиента: загруженные документы + финально утвержденная декларация.
- Формировать обучающие примеры вида: `(контекст документов, описание товара) -> (утвержденный код ТН ВЭД)`.
- Использовать примеры в ранжировании кандидатов в `ai-service` (RAG + rule-based + precedent score).
- Добавить механизм валидации качества (offline eval на отложенной выборке до включения в production).

#### 1.5 Учет истории кодов ТН ВЭД клиента

- Хранить историю примененных кодов ТН ВЭД в разрезе клиента, контрагента, товарного описания и периода.
- Учитывать историю как отдельный сигнал при подборе кода (вес клиента выше глобального прецедента).
- В UI показывать объяснимость: "код предложен на основе N прошлых деклараций клиента".
- Добавить контроль drift: если новый код конфликтует с устойчивой историей, показывать предупреждение и запрашивать подтверждение.

#### Definition of Done (Фаза 1)

- Можно отправить email с документами и получить созданный черновик декларации без входа в UI.
- В админке можно создать/изменить AI-стратегию, и она реально влияет на итоговый JSON.
- Пользователь может сверять поля декларации с исходным документом в одном экране.
- Подбор ТН ВЭД улучшается на исторических данных без деградации качества на контрольной выборке.
- Пользователь видит источник рекомендации кода (документ/история клиента/прецедент).

### Фаза 2: Trade Compliance и финансовый контроль

Цель: снизить риск штрафов и ошибок по регуляторным требованиям.

#### 2.1 Модуль санкционного комплаенса

- Создать `compliance-service`.
- Интегрировать открытые источники санкционных списков (OFAC, EU Sanctions, РФ).
- Автопроверка:
  - контрагентов (наименование, ИНН/Tax ID),
  - кодов ТН ВЭД на признаки двойного назначения.
- Выводить предупреждения (alerts) в UI декларации.

#### 2.2 Валютный контроль (УНК и платежные документы)

- Расширить `ai-service`:
  - `unk_parser.py` (номер УНК, банк, суммы),
  - `payment_order_parser.py` (платежные поручения).
- Сверять суммы: invoice vs УНК vs платежки.
- Показывать баланс/расхождения по контракту в интерфейсе.

#### 2.3 Оплата услуг через счет ТБанка

- Добавить модуль выставления счета на оплату услуг брокера через ТБанк.
- Поддержать генерацию счета, статус оплаты и webhook-подтверждения от ТБанка.
- Связать оплату с сущностями клиента/декларации/услуги (invoice_id, declaration_id, client_id).
- В UI показывать статус: "счет выставлен / частично оплачен / оплачен / просрочен".

#### Definition of Done (Фаза 2)

- Для декларации видно compliance-статус и причины риска.
- В системе есть единый финансовый блок сверки по УНК/платежам/инвойсу.
- Критичные расхождения и санкционные риски блокируют авто-применение без подтверждения пользователя.
- Счета ТБанка создаются и обновляются автоматически, статусы оплаты синхронизируются по webhook.

### Фаза 3: Цифровой логист (Supply Chain)

Цель: расширить ценность продукта в логистический контур и увеличить конверсию в доп.услуги.

#### 3.1 ИИ-тарификатор (Smart Quoting)

- Парсить ставки морских и ж/д перевозчиков из Excel/PDF в структурированную базу.
- Добавить интерфейс запроса ставки: параметры груза и маршрута -> 3 варианта (быстро/дешево/оптимально).

#### 3.2 Оптимизатор загрузки (Load Planner)

- Использовать данные packing list (вес, объем, габариты).
- Считать потребность в контейнерах (20', 40'HQ) или фурах.
- Выдавать расчет утилизации (текстом или с простой визуализацией).

#### 3.3 Генератор транспортных документов

- По данным инвойса и packing list автогенерировать CMR, AWB, коносамент в PDF.
- Технологии генерации: WeasyPrint или ReportLab.

#### Definition of Done (Фаза 3)

- Можно получить коммерческое предложение по перевозке из формы запроса.
- Система выдает расчет загрузки и документирует принятый вариант.
- Транспортные документы генерируются автоматически из данных сделки.

### Фаза 4: Интеграции и SaaS-метрики

Цель: встроиться в контуры клиентов и обеспечить прозрачную монетизацию.

#### 4.1 Экспорт в Альта-ГТД / СТМ

- Развивать `integration-service`.
- Сделать шаблоны маппинга (`Jinja2`) из внутреннего JSON декларации в целевые XML-форматы.
- Давать скачивание готовых файлов для импорта в ПО клиента.
- Предусмотреть генерацию XML-пакета для передачи в ФТС РФ: декларация + структурированные XML приложенных документов (инвойс, упаковочный лист, контракт и др. по профилю обмена).
- Добавить версионирование XML-схем и контроль совместимости форматов.

#### 4.2 Интеграция с 1С

- Генерировать XML/JSON в формате EnterpriseData (1С:ERP / 1С:Бухгалтерия).
- Передавать данные по выпущенным товарам в учетные системы клиента.

#### 4.3 Cost Governance (биллинг AI)

- Считать input/output токены по каждой декларации.
- Сохранять стоимость обработки в БД.
- Добавить админ-дашборд unit-экономики (стоимость обработки 1 декларации).

#### 4.4 Telegram-бот для клиентов

- Реализовать на `aiogram` (или совместимой платформе).
- Функции: отправка документов, статусы декларации, уведомления о выпуске, запрос недостающих данных.

#### 4.5 ЭЦП (электронная цифровая подпись)

- Добавить поддержку ЭЦП для юридически значимого подписания пакета документов.
- Интегрировать подписание в workflow отправки (проверка сертификата, сроков действия, статуса подписи).
- Хранить audit trail: кто подписал, когда подписал, какой сертификат использован, хэш документа.
- Поддержать верификацию подписи при повторном открытии/экспорте документов.

#### 4.6 Интеграция с ФТС РФ (информационное взаимодействие)

- Реализовать контур подключения платформы к ФТС РФ для передачи декларации и приложений в утвержденных форматах.
- Добавить транспортный адаптер обмена: отправка пакета, получение квитанций/статусов, обработка ошибок протокола.
- Поддержать жизненный цикл сообщений: "отправлено", "принято", "отклонено", "требуется корректировка", "выпуск разрешен/завершен".
- Реализовать журнал обмена с ФТС (trace-id, timestamp, payload hash, код ответа, текст ошибки).
- Обеспечить повторную отправку (retry) и идемпотентность операций.

#### Definition of Done (Фаза 4)

- Клиент может использовать платформу через внешние системы (таможенное ПО, 1С, Telegram) без ручного дублирования данных.
- У администратора есть прозрачная аналитика затрат на AI по клиентам и декларациям.
- Ключевые документы могут подписываться ЭЦП и проходить проверку валидности подписи.
- Система формирует XML-пакет для ФТС РФ (декларация + приложения) с успешной валидацией по схеме.
- Реализован сквозной обмен с ФТС: отправка, получение статусов, сохранение квитанций и обработка отклонений.

## Приоритеты запуска

1. Фаза 1 (критический путь, ближайший релиз).
2. Фаза 2 (регуляторные риски, финансовая консистентность и платежный контур ТБанка).
3. Фаза 4.3 (Cost Governance) как сквозная задача для контроля экономики AI уже с Фазы 1.
4. Фаза 4.5 (ЭЦП) как обязательный шаг для юридически значимого документооборота.
5. Фаза 3 и остальные интеграции Фазы 4 — по мере стабилизации core-потока.

## Технические предпосылки

- Для Фазы 1 нужен брокер сообщений для асинхронной обработки (Redis Pub/Sub или RabbitMQ).
- Потребуется увеличение оперативной памяти сервера с ростом фоновых задач и AI-нагрузки.
- Все новые сервисы должны публиковать health-check и метрики для централизованного мониторинга.

## Глоссарий (кратко)

- Intake: канал первичного входа документов в систему.
- Compliance: проверка соответствия санкционным и регуляторным требованиям.
- DoD (Definition of Done): измеримые критерии, по которым задача/фаза считается завершенной.
- Unit-экономика: стоимость обработки одной бизнес-единицы (здесь: 1 декларации).
- ЭЦП: электронная цифровая подпись для подтверждения юридической значимости документа.
