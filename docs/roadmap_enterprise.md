# Roadmap: Enterprise Customs Automation Platform

Единый план развития платформы от текущего AI-парсера деклараций к полноценной экосистеме ВЭД: документы, комплаенс, финансы, логистика, интеграции и клиентский сервис.

## Как использовать этот документ

- Это мастер-план (стратегический уровень).
- Каждая фаза детализируется в отдельный план реализации (тактический уровень): `docs/plans/<phase>-<topic>.md`.
- Все новые feature-планы должны ссылаться на конкретный пункт этого roadmap.
- Изменения статусов фаз вносятся сначала в детальные планы, затем агрегируются сюда.

## Целевой продукт

Платформа автоматизации таможенного оформления, которая:
- принимает документы из разных каналов (web, email, bot),
- автоматически формирует и проверяет декларацию,
- снижает регуляторные и финансовые риски,
- поддерживает self-service оплату услуг в личном кабинете через Т-Банк (привязка карты или счет на оплату),
- добавляет смежные сервисы (цифровой логист, интеграции с клиентскими системами),
- рассчитывает стоимость дороги, сроки и оптимальные мультимодальные маршруты,
- поддерживает контроль unit-экономики AI-обработки.

## Продуктовый принцип приоритизации

- Сначала доводим до конца **цифрового брокера таможенных деклараций**: удобный intake, корректная работа core flow, AI-помощник, оплата, юридически значимая отправка и прозрачный статус кейса.
- Все новые функции оцениваются по правилу: уменьшают ли они ручную работу брокера, повышают ли качество декларации, улучшают ли UX клиента или ускоряют оплату/запуск кейса.
- Смежные контуры (`цифровой логист`, расширенная мультиагентность, дополнительные SaaS-модули) развиваются после стабилизации и монетизации брокерского ядра.

## Реализованный базовый функционал (foundation)

Перечень возможностей, реализованных до начала работы по фазам роадмапа:

- **Микросервисная архитектура:** core-api (8001), file-service (8002), ai-service (8003), integration-service (8004), calc-service (8005).
- **Frontend:** React 18 + TypeScript + MUI 6 + TanStack Query, 14 страниц, 12+ компонентов.
- **Auth + RBAC:** JWT + bcrypt, 7 ролей, company filter middleware, AuditLog.
- **CRUD:** декларации, позиции, документы, контрагенты, компании, пользователи.
- **AI-парсинг:** smart_parser (инвойсы, контракты, упаковочные, спецификации, транспорт), CrewAI/линейный pipeline. Расчёт графы 42 (фактурная стоимость), маппинг техописания к позициям, привязка весов из packing list.
- **RAG ТН ВЭД:** ChromaDB, 39K+ кодов, DSPy классификация, feedback/precedents.
- **Оценка рисков:** risk_engine, risk_rules.json, RAG по правилам СУР, DSPy RiskAnalyzer.
- **Rules Engine:** `declaration_mapping_v3.yaml`, `graph_rules` (CRUD API + миграция).
- **Файловое хранилище:** MinIO через file-service.
- **Расчет платежей:** calc-service, курсы ЦБ (Redis cache).
- **XML-экспорт:** integration-service (базовый xml_builder + xml_validator).
- **Админка:** дашборд, пользователи, аудит, база знаний, чеклисты.
- **CI/CD:** GitHub Actions (deploy-backend.yml, deploy-frontend.yml).
- **Logging:** structlog + JSON во всех сервисах.
- **Docker:** docker-compose.yml (dev) + docker-compose.prod.yml (prod), Nginx reverse proxy.

## Фазы развития

### Фаза 0: Инфраструктура и масштабирование (Подготовка к 1000 ДТ/день)
*Цель: Обеспечить отказоустойчивость платформы при пиковых нагрузках, подготовить инфраструктуру к внедрению ресурсоемких агентов "Цифрового логиста".*

Простыми словами, Фаза 0 нужна, чтобы цифровой брокер:
- не падал, когда одновременно загружают много документов;
- не зависал на тяжелом AI-парсинге и фоновых задачах;
- быстро диагностировался по логам, метрикам и `trace-id`;
- переживал неудачный релиз, сбой сервиса или потерю узла без потери данных;
- мог безопасно расти до более дорогих AI-функций, оплаты и интеграций.

**Важно:** Фаза 0 не должна тормозить развитие пользовательских функций. Для ближайших релизов берем из нее только то, что напрямую защищает core flow цифрового брокера.

1. **Переход на production-сервер и базовый deploy-контур** - *ЧАСТИЧНО СДЕЛАНО*
   - **(СДЕЛАНО)** Проект уже развернут на production-сервере, backend/frontend деплоятся по SSH/GitHub Actions.
   - **(В ОЧЕРЕДИ)** Подтвердить sizing сервера под целевую нагрузку:
     - **CPU:** 4 – 6 ядер (vCPU) минимум (рекомендуется 8-12 для параллельных агентов).
     - **RAM:** 16 ГБ минимум (рекомендуется 32 ГБ для параллельного OCR и тяжелых RAG).
     - **Disk:** от 50 ГБ NVMe / SSD (100+ ГБ для хранения PDF архивов и базы ChromaDB).
   - **(В ОЧЕРЕДИ)** Настройка swap-файла, Docker limits/reservations и capacity planning.

2. **Очередь задач (Task Queue & Workers)** - *В ОЧЕРЕДИ*
   - Внедрение Celery или ARQ (для асинхронного Python) + RabbitMQ / Redis.
   - Вынос тяжелого AI-парсинга и CrewAI агентов в фоновые воркеры.
   - Это предотвратит падение `ai-service` по OOM (Out Of Memory) или тайм-ауту при одновременной загрузке десятков пакетов документов.
   - Масштабирование: запуск нескольких инстансов `ai-service` worker для параллельной обработки.
   - Ввести отдельный класс задач `agent_jobs`: reviewer, reconciliation, compliance, logistics-planner.
   - Добавить retry policy, dead-letter queue и ручную очередь разбора для упавших агентных задач.

3. **SRE-модель (SLI/SLO и incident response)** - *В ОЧЕРЕДИ*
   - Зафиксировать SLI: availability, p95/p99 latency, queue lag, error rate.
   - Утвердить SLO и error budget по критичным сервисам (`core-api`, `ai-service`, `integration-service`).
   - Внедрить on-call процесс и runbook для P1/P2 инцидентов.
   - Проводить postmortem без обвинений и game day для проверки готовности.

4. **Observability и журналирование** - *ЧАСТИЧНО СДЕЛАНО*
   - **(СДЕЛАНО)** structlog + JSON во всех 5 сервисах (core-api, ai-service, file-service, integration-service, calc-service).
   - **(СДЕЛАНО)** `LoggingMiddleware` в core-api: генерация/чтение `X-Request-ID`, привязка `correlation_id` к structlog context.
   - **(СДЕЛАНО)** `TracingMiddleware` добавлен в ai-service, file-service, integration-service, calc-service — сквозной `X-Request-ID`.
   - **(СДЕЛАНО)** Nginx генерирует `X-Request-ID` (`$request_id`) и пробрасывает во все backend-сервисы.
   - **(СДЕЛАНО)** Frontend добавляет `X-Request-ID` через `crypto.randomUUID()` в axios interceptor.
   - **(СДЕЛАНО)** Все inter-service httpx-вызовы (core-api → ai/calc/file, ai-service → core-api) передают `X-Request-ID`.
   - **(В ОЧЕРЕДИ)** Централизовать логи, метрики и трассировку (ELK/Loki/Grafana stack).
   - **(В ОЧЕРЕДИ)** Добавить дашборды по 4 golden signals и алерты по SLO.

5. **DR/BCP (резервирование и восстановление)** - *В ОЧЕРЕДИ*
   - Настроить backup и PITR для PostgreSQL, backup для MinIO/ChromaDB.
   - Утвердить RPO/RTO для критичных контуров (декларации, платежи, обмен с ФТС).
   - Проводить регулярные restore-тесты и фиксировать результат в журнале.

6. **Security Baseline (секреты и доступы)** - *В ОЧЕРЕДИ*
   - Хранить секреты только во внешнем secret manager (Vault/KMS), исключить хранение в git/образах.
   - Ввести ротацию ключей/токенов (не реже 1 раза в 90 дней).
   - Включить аудит доступа к секретам и критичным операциям.

7. **AppSec в CI (Secure SDLC)** - *В ОЧЕРЕДИ*
   - На каждый PR запускать SAST, SCA, secret scan и container scan.
   - Генерировать SBOM для релизных артефактов.
   - Блокировать merge/release при high/critical уязвимостях без утвержденного waiver.

8. **Единый CI/CD для всех сервисов** - *ЧАСТИЧНО СДЕЛАНО*
   - **(СДЕЛАНО)** GitHub Actions: `deploy-backend.yml`, `deploy-frontend.yml`.
   - **(В ОЧЕРЕДИ)** Единый pipeline: build -> test -> scan -> deploy для всех сервисов.
   - **(В ОЧЕРЕДИ)** Staged rollout (canary/blue-green) и проверяемый rollback.
   - **(В ОЧЕРЕДИ)** Smoke-тесты после деплоя и health-gate перед переводом в production.

#### Приоритет внутри Фазы 0

**Сначала делаем (блокирует цифрового брокера уже сейчас):**
- очередь задач и фоновые worker-ы для тяжелого AI;
- smoke-тесты после деплоя и health-gate;
- backup/restore для PostgreSQL, MinIO и критичных AI-артефактов;
- базовую observability: централизованные логи, метрики, трассировку, поиск по `trace-id`.

**Потом делаем (повышает зрелость платформы, но не должно тормозить broker core):**
- формальные SLI/SLO, on-call и incident response;
- внешний secret manager, ротацию ключей и расширенный аудит доступов;
- полный AppSec-контур в CI (`SAST`, `SCA`, `SBOM`, container scan);
- staged rollout, canary/blue-green и прочие enterprise-практики зрелого deploy-пайплайна.

#### Definition of Done (Фаза 0)

- Нагрузочный тест подтверждает устойчивую обработку целевого потока без деградации SLO.
- Для критичных сервисов действуют SLI/SLO, алерты и on-call с runbook.
- В production есть централизованные логи/метрики/трейсы и сквозной `trace-id`.
- Регулярный restore-тест backup проходит успешно в рамках утвержденных RPO/RTO.
- CI/CD с AppSec-гейтами покрывает все сервисы и поддерживает быстрый rollback.

### Фаза 1: Цифровой брокер: intake, UX и AI-ядро (Абсолютный приоритет)

Цель: довести до конца основной продуктовый контур цифрового брокера: быстрый старт кейса, удобная работа пользователя, корректный AI-черновик декларации, прозрачные ошибки и стабильный end-to-end flow.

- **(В ОЧЕРЕДИ)** Полная RBAC-матрица с разрешениями на уровне операций.

#### 1.7 Pre-send hard gate (перед отправкой) - *ЧАСТИЧНО СДЕЛАНО*

- **(СДЕЛАНО)** Серверная функция `run_pre_send_checks()` — 8 категорий проверок:
  - Обязательные поля декларации (type_code, company_id, страны, incoterms, валюта, стоимость).
  - Наличие отправителя и получателя (графы 2, 8).
  - Наличие позиций товаров (графа 31/33).
  - Наличие прикреплённых документов.
  - Позиции без кода ТН ВЭД.
  - Нерешённые блокирующие `ParseIssue` в БД.
  - Блокирующие AI-проблемы из `ai_issues` JSONB.
  - Весовые аномалии (нетто > брутто).
- **(СДЕЛАНО)** Endpoint `GET /declarations/{id}/pre-send-check` — клиентская проверка до перехода.
- **(СДЕЛАНО)** Блокировка перехода в `signed`/`sent` при blocking issues (HTTP 422).
- **(СДЕЛАНО)** Override с `force=true` + обязательный `override_reason` + запись в `DeclarationLog` (audit trail).
- **(СДЕЛАНО)** Клиентский `DeclarationChecklist` компонент (11 проверок на фронте).
- **(ЧАСТИЧНО СДЕЛАНО)** Базовая матрица обязательных документов: `invoice`, `contract`, `transport_doc`, `packing_list` уже проверяются в `pre-send`.
- **(ЧАСТИЧНО СДЕЛАНО)** Базовая cross-doc сверка уже добавлена: сумма декларации vs инвойс, валюта декларации vs договор/инвойс, веса и места vs packing list, число позиций vs invoice, наличие transport doc без переноса реквизитов.
- **(В ОЧЕРЕДИ)** Расширить матрицу обязательных документов по сценарию (импорт/экспорт, процедура, Incoterms, отраслевые кейсы).
- **(В ОЧЕРЕДИ)** Довести cross-doc сверку до полного reconciliation-уровня (УНК, платежки, доп. документы, более строгие бизнес-правила).

#### 1.8 AI Evals и релизный quality gate - *ЧАСТИЧНО СДЕЛАНО*

- **(СДЕЛАНО)** Зафиксировано правило: файлы `GTD*.pdf` в `dek/` трактуются как эталонные готовые декларации (`golden/reference outputs`), а не как входные документы для `parse-smart`.
- **(В ОЧЕРЕДИ)** Собрать offline eval/regression набор по критичным графам и подбору ТН ВЭД на базе `dek/` + эталонных `GTD*`.
- **(В ОЧЕРЕДИ)** Запускать eval в CI после изменений промптов, моделей, индекса или правил.
- **(В ОЧЕРЕДИ)** Блокировать релиз при деградации метрик ниже согласованного baseline.

#### 1.9 AI Control Plane и agent escalation layer

- Зафиксировать принцип: **deterministic core first** — основной `parse-smart` остается быстрым линейным pipeline, агенты подключаются только по условиям эскалации.
- Условия запуска агентного слоя:
  - низкий `confidence`,
  - конфликт данных между документами,
  - нестыковка с историей компании,
  - high-risk кейс,
  - провал `pre-send hard gate`,
  - неполный пакет документов.
- Первый набор специализированных агентов:
  - `ReconciliationAgent` — сверка `invoice vs packing list vs contract vs transport docs`,
  - `ReviewerAgent` — независимая перепроверка финального JSON и `evidence_map`,
  - `ClientQuestionAgent` — формирование точного запроса клиенту на недостающие документы/поля.
- Контракт результата агента должен быть структурированным: `decision`, `confidence`, `evidence`, `issues[]`, `rule_version`, `precedent_ids`, `next_action`.
- Для каждого agent run фиксировать стоимость, latency, trace-id, причину запуска и итоговое влияние на решение.

#### 1.10 Autopilot Intake и запрос недостающих данных

- После первичного `parse-smart` автоматически определять, каких документов/полей не хватает для продолжения.

### Фаза 3: Цифровой логист (Supply Chain & Routing)

Цель: следующий большой продуктовый контур после завершения ядра цифрового брокера — модуль, который считает дорогу, сроки, оптимальный маршрут, загрузку и пакет транспортных документов.

#### 3.1 ИИ-тарификатор (Smart Quoting)

- Парсить ставки морских, ж/д, авто- и авиаперевозчиков из Excel/PDF/API в структурированную базу.
- Нормализовать базовые ставки, surcharges, терминальные сборы, last mile, сезонные и пограничные надбавки.
- Добавить интерфейс запроса ставки: параметры груза и маршрута -> 3 варианта (`быстро / дешево / оптимально`).

#### 3.2 Расчет дороги и оптимизатор маршрута (Route Optimizer)

- Считать полный `door-to-door` маршрут: плечи, тип транспорта, ETA, точки перевалки, переходы границы, риски и ограничения.
- Поддержать мультимодальные сценарии: море + ж/д + авто, авиа + авто, прямой авто и другие типовые цепочки.
- Строить 3 класса рекомендаций:
  - минимальная стоимость,
  - минимальный срок,
  - лучший компромисс цена/срок/риск.
- Учитывать Incoterms, страну отправления/назначения, характеристики груза, температурный режим, DG/ADR-ограничения, тип клиента и SLA.
- Показывать breakdown стоимости дороги: фрахт, терминалы, склад, страхование, last mile, таможенные платежи, агентские услуги.

#### 3.3 Оптимизатор загрузки (Load Planner)

- Использовать данные packing list (вес, объем, габариты).
- Считать потребность в контейнерах (20', 40'HQ) или фурах.
- Выдавать расчет утилизации, сценарии консолидации и рекомендации по перепаковке/разбиению партии.

#### 3.4 Генератор транспортных документов

- По данным инвойса и packing list автогенерировать CMR, AWB, коносамент в PDF.
- Технологии генерации: WeasyPrint или ReportLab.
- Привязывать документы к выбранному маршруту и автоматически подставлять фактические логистические параметры.

#### Definition of Done (Фаза 3)

- Можно получить коммерческое предложение по перевозке из формы запроса с разбивкой стоимости дороги.
- Система выдает минимум 3 маршрута (`быстро / дешево / оптимально`) с ETA, рисками и обоснованием выбора.

## Фаза 5: Мультиагентная архитектура (Контролируемая оркестрация)
*Цель: превратить текущий AI pipeline в управляемую мультиагентную систему без потери предсказуемости, explainability и unit-экономики.*

### 5.1 AI Control Plane и политики запуска агентов
- Ввести единый оркестратор agent-run задач: registry агентов, versioning, budgets, retries, dead-letter handling.
- Агенты запускаются только по политикам, а не на каждый `parse-smart`.
- Базовые триггеры запуска:
  - low confidence,
  - конфликт документов,
  - высокорисковый кейс,
  - fail pre-send gate,
  - нестандартный маршрут/логистический сценарий,
  - аномальная стоимость или отклонение от истории.
- Для каждого agent run фиксировать `trace-id`, стоимость, latency, входные артефакты, итоговое влияние на решение.
- Контракт результата всех агентов унифицировать: `decision`, `confidence`, `evidence`, `issues[]`, `next_action`, `human_review_required`.

### 5.2 Линейка специализированных агентов
- `TriageAgent` — быстрая классификация входящих документов и маршрутизация кейса.
- `ReconciliationAgent` — сверка документов сделки и поиск финансовых/весовых/количественных конфликтов.
- `HSReviewerAgent` — независимая перепроверка спорных кодов ТН ВЭД и отклонений от истории компании.
- `ComplianceGuardianAgent` — санкции, dual-use, правила СУР, критичные регуляторные триггеры.
- `LogisticsPlannerAgent` — подбор маршрута, расчет стоимости дороги, оценка рисков доставки и сроков.
- `ClientCopilotAgent` — формирование точных запросов клиенту, пояснений и статусов в ЛК / Telegram / email.
- `FilingAgent` — подготовка пакета к отправке после прохождения hard gate и сверки evidence package.

### 5.3 Event-Driven runtime и human-in-the-loop
- Перевод агентного слоя из встроенного вызова в асинхронную шину событий.
- Поддержать fan-out/fan-in сценарии: несколько агентов анализируют один кейс, оркестратор собирает результат.
- Добавить review queue для человека: агент не применяет критичные изменения без явного approval.
- Все override-решения человека возвращать в память системы как feedback и precedent.

### 5.4 Precedent Graph / Cortex Memory
- Развить текущие прецеденты в структурированную память: `company + supplier + HS + route + customs_office + outcome`.
- Хранить не только похожие описания, но и связь с исходом: выпуск, запрос допдоков, отказ, override брокером.
- Использовать память в `ReconciliationAgent`, `HSReviewerAgent` и `LogisticsPlannerAgent` как слой explainable recommendations.

#### Definition of Done (Фаза 5)

- Агентный слой управляется оркестратором и не запускается без политики/триггера.
- Каждый agent run полностью трассируется: стоимость, latency, trace-id, версия агента, итоговое решение.
- Критичные решения проходят через human-in-the-loop и оставляют audit trail.
- Измеримо сокращается ручная работа брокера по конфликтным кейсам, сверке и коммуникации с клиентом.

## Приоритеты запуска

1. **Сначала закрываем ядро цифрового брокера**: Фаза 1 целиком (`intake`, UX, `evidence_map/issues[]`, pre-send hard gate, AI quality, Autopilot Intake, e2e regression).
2. **Параллельно берем из Фазы 0 только блокирующие reliability-задачи**: очередь задач, smoke tests, backup, observability, health-gates. Переход на сервер как отдельная цель больше не является ближайшим приоритетом.
3. **Сразу после этого закрываем денежный контур брокера**: Фаза 2.2 и 2.3 — финансовая сверка, УНК/платежки, оплата в личном кабинете через Т-Банк.
4. **Следом завершаем юридически значимый контур цифрового брокера**: 4.1 (экспорт в Альта/СТМ), 4.5 (ЭЦП), 4.6 (ФТС), 4.8 (Compliance Evidence Package).
5. **Сквозной AI quality gate и Cost Governance обязательны с первых релизов**: regression eval, dependency governance, контроль стоимости `agent runs` и стоимости 1 декларации.
6. **Phase 3 / цифровой логист — следующий горизонт роста**, только после того как core-брокер стабилен, удобен пользователю и монетизируется.
7. **Фаза 5 / полноценная Event-Driven мультиагентность — после стабилизации брокерского ядра**, а не раньше.

## Три очереди работ

### Очередь 1: Must-have сейчас

Это все, без чего цифровой брокер еще нельзя считать завершенным как основной продукт.

- `Фаза 1.1` Email / webhook intake и автосоздание черновика декларации.
- `Фаза 1.4` Автоформирование обучающих примеров + offline eval по ТН ВЭД.
- `Фаза 1.5` Контроль drift по кодам ТН ВЭД клиента.
- `Фаза 1.7` Матрица обязательных документов и полноценная cross-doc сверка.
- `Фаза 1.8` Регрессионный AI quality gate в CI.
- `Фаза 1.9` Базовый agent escalation layer только для конфликтных кейсов.
- `Фаза 1.10` Autopilot Intake: автоопределение недостающих документов и запрос клиенту.
- `Фаза 1.11` UX цифрового брокера: upload wizard, status timeline, diff-view, единый центр проблем.
- `Фаза 1.12` Надежность core flow: e2e regression на `dek/`, post-deploy smoke tests, retry/idempotency, нормальные ошибки.
- `Фаза 0` в урезанном обязательном виде: очередь задач, backup/restore, observability, health-gates.

### Очередь 2: После запуска оплаты

Это следующий слой зрелости, когда core flow уже стабилен и клиент может сам платить в личном кабинете.

- `Фаза 2.2` Валютный контроль: УНК, платежки, explainable reconciliation сумм.
- `Фаза 2.3` Оплата через Т-Банк: привязка карты, счет/платежная ссылка, webhook-статусы, связка с декларацией и услугой.
- `Фаза 2.1` Санкционный комплаенс и проверка dual-use.
- `Фаза 2.5` Версионирование compliance-правил и rollback.
- `Фаза 4.1` Доведение экспорта в Альта/СТМ до production-ready XML.
- `Фаза 4.5` ЭЦП.
- `Фаза 4.6` Контур отправки и статусов ФТС.
- `Фаза 4.8` Compliance Evidence Package.
- `Фаза 4.9` Notification-service для blocker issues, оплат, reject-ов и статусов.

### Очередь 3: После стабилизации broker core

Это уже масштабирование продукта за пределы ядра цифрового брокера.

- `Фаза 3` Цифровой логист: ставки, стоимость дороги, оптимальные маршруты, load planner, транспортные документы.
- `Фаза 5` Полноценная Event-Driven мультиагентность и agent control plane как отдельный зрелый слой.
- `Фаза 4.2` Интеграция с 1С.
- `Фаза 4.4` Telegram-бот как полноценный клиентский канал, а не только транспорт документов.
- `Фаза 4.7` Продвинутый multi-tenant SaaS: квоты, tenant billing, tenant isolation tests.
- Полный enterprise-контур из `Фазы 0`: SLI/SLO, on-call, AppSec, secret manager, canary/blue-green.

## Технические предпосылки

- Для Фазы 0 и 1 нужен брокер сообщений для асинхронной обработки (Redis Pub/Sub или RabbitMQ).
- Потребуется увеличение оперативной памяти сервера с ростом фоновых задач и AI-нагрузки (мин. 16ГБ RAM).
- Все новые сервисы должны публиковать health-check и метрики для централизованного мониторинга.
- Для всех сервисов обязательны структурированные startup/runtime логи с `trace-id` для поиска ошибок запуска и работы.
- Для AI-фич обязательна синхронизация кода и зависимостей (`requirements`/lockfile + CI проверка импортов) без скрытых fallback.
- Изменения моделей/промптов/индексов допускаются только после прохождения regression eval.
- Для агентного слоя нужны единые схемы task/result payload и явные лимиты `cost / latency / retry`.
- Для цифрового логиста нужны нормализованные тарифные источники, справочник маршрутов/узлов и модель мультимодальных плеч.
- Для оплаты через Т-Банк нужны безопасные webhook-и, idempotency keys и хранение только токенизированных payment method reference.

## Глоссарий (кратко)

- Intake: канал первичного входа документов в систему.
- Compliance: проверка соответствия санкционным и регуляторным требованиям.
- DoD (Definition of Done): измеримые критерии, по которым задача/фаза считается завершенной.
- Unit-экономика: стоимость обработки одной бизнес-единицы (здесь: 1 декларации).
- ЭЦП: электронная цифровая подпись для подтверждения юридической значимости документа.
